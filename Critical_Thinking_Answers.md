1) So basically when I was setting up Firebase in my main.dart, I had to add WidgetsFlutterBinding.ensureInitialized() right at the top of the main function before anything else. The reason is that Firebase.initializeApp() needs to talk to native Android code to set things up, and Flutter needs to have its binding ready before that can happen. I actually forgot to add it at first and the app just crashed on launch with some binding error. Once I added it back everything worked fine. Without it Flutter doesn't know how to communicate with the platform side yet so any async native calls will just fail.

2) In my app I used StreamBuilder to show the product list because it listens to changes in real time. So whenever I added a new product or deleted one from the app or even from the Firebase console directly, the list updated automatically without me having to refresh anything. A FutureBuilder would only fetch the data once when the widget loads and then that's it. If I used FutureBuilder and added a product, I'd have to manually trigger another fetch to see it show up. StreamBuilder made way more sense here because the whole point of the app is managing products in real time. I'd probably only use FutureBuilder for something like loading a user profile once when you open a settings page.

3) Firestore is NoSQL because instead of having tables with strict rows and columns like in MySQL or something, it uses collections and documents. Each document is basically like a JSON object with key-value pairs. So in my products collection, each document has a name and price but I could easily add a category field to one document without changing every other document. In a SQL database I'd need to alter the whole table schema to add a column. For mobile apps this flexibility is really nice because I don't need to worry about migrations or schemas when I want to add new features. Plus Firestore syncs data in real time and works offline which is perfect for mobile.

4) When I set up Firestore I chose test mode which basically means anyone can read or write anything. That's fine for this class activity but if this was a real app it would be a huge security problem because literally anyone who finds my Firebase config could delete all the products or add garbage data. For a real product management app I would set up rules so that only logged-in users can read products and only admin users can create, update, or delete them. Something like checking if request.auth is not null for reads and then checking if the user has an admin role for writes.

5) Right now in my app I only check if the name field isn't empty and then I just parse whatever is in the price field as a double. I realized that's kind of risky because someone could enter a negative price or type something weird and it would either crash or save bad data. I should add checks to make sure the price is a positive number, maybe set a max length on the product name so nobody enters a super long string, and also check that the price isn't something crazy high. Input validation is important because once bad data gets into the database it messes up everything. Like if a price somehow got saved as -50 my price filter wouldn't work right and the app would look broken.

6) Right now my StreamBuilder is listening to the entire products collection which works fine because I only have a few test products. But if there were 10000 products it would be really slow because it would try to load all of them at once and also every time anything changes it would re-download data. That would also eat through the Firestore read quota fast. To fix this I could use pagination with the limit() function so I only load like 20 products at a time and then load more as the user scrolls. I could also add Firestore indexes on fields like price so that queries with where clauses run faster. And doing the filtering on the server side with Firestore queries instead of downloading everything and filtering in the app would save a lot of bandwidth.

7) When I use snapshots() in my code it opens a persistent connection between my app and Firestore. Whenever a document in the products collection changes Firestore sends just the changed data to my app through that connection and StreamBuilder automatically rebuilds the UI. The benefit is that everything updates instantly without needing a refresh button which makes the app feel really responsive. But the downside is that Firestore charges per document read so every change counts as a read for every user listening. If I had 100 users all looking at the product list and someone updated one product that's 100 reads right there. It also uses more battery since the connection stays open the whole time. For my class project it's totally fine but for a production app with lots of users I'd need to think about cost.

8) The firebase_options.dart file has all the configuration stuff my app needs to connect to the right Firebase project like the API key, project ID, and app ID for each platform. When I ran flutterfire configure it automatically generated this file for me. It's auto-generated because manually typing in all those long strings from the Firebase console would be super error prone. One wrong character in the API key and the app wouldn't connect at all. The CLI just reads everything directly from my Firebase project and writes the correct values so there's no chance of me messing it up. It also handles different platforms automatically so if I ever added iOS support it would just add those config values too.

9) If I wanted to make the app more useful I'd probably add fields like category, stock, and maybe an image URL. To do that I'd first update the bottom sheet form in my createOrUpdate function to have extra TextFields for each new field. Then I'd update the products.add() and products.doc().update() calls to include those new fields in the map. For the UI I'd change the ListTile to show the category maybe as a little chip under the product name and use Image.network() if there's an image URL. The cool thing about Firestore being NoSQL is that I don't need to change the database structure at all. My old products that only have name and price would still work fine and the new ones would just have the extra fields.

10) So I found out that Firestore actually has offline persistence built in and it's enabled by default on Android. When the app loses internet connection Firestore caches all the data locally so users can still see products they loaded before. If someone adds or edits a product while offline those changes get saved locally and then automatically synced back to the server when the internet comes back. The app doesn't even need to handle this manually because StreamBuilder still works with the cached data. To make it even better I could add a little banner at the top of the screen that shows when the app is offline so users know the data might not be current. I could also set the cache size to unlimited with FirebaseFirestore.instance.settings so it doesn't start clearing old data.
